# 07. 리캐치(Re:catch) 스타일 가이드

> **소스 1 (웹):** 리캐치 자체 콘텐츠 방법론("재·쓰·비" 프레임워크), 2026 네이버 검색 알고리즘(C-Rank / D.I.A+)
> **소스 2 (데이터):** `recatch_blog_raw.csv`(245개 글) + `recatch_blog_ai_crm.csv`(24개 글)의 구조·문체 정량 분석
> **용도:** 네이버 블로그 글 생산 파이프라인(writer·image·assembler 에이전트) 및 사람 작성자용 작성 표준

## Part 0. 리캐치 스타일이란 — 한 줄 정의

> **"광고가 아니라 인사이트로 읽히는, B2B 실무자 동료의 글."**

제품 홍보 글인데도 독자가 "배웠다"는 느낌으로 읽고 광고로 인식하지 않게 만드는 방식입니다. 리캐치는 이 방식으로 **블로그 월 트래픽 2배, 작성 시간 68% 감소, 생산성 3.14배**를 달성했다고 공개했습니다.

### 리캐치가 직접 밝힌 핵심 철학
- AI가 **'잘 쓴 글'**은 누구나 만들 수 있지만, **'좋은 글'**(사람이 읽고 사랑하는 글)은 **사람의 기획·관점**이 있어야 한다.
- 그래서 리캐치는 "AI를 안 쓰는 것"이 아니라 **"AI를 잘 쓸 결심"**을 택했고, 도구(ChatGPT→Claude)와 프레임워크(재·쓰·비)를 처음부터 다시 설계했다.
- 즉 리캐치 스타일의 본질은 **표현 기교가 아니라 '기획·관점·증거'**에 있다.

## Part 1. 리캐치 스타일의 7대 DNA

CSV 245개 글에서 반복적으로 검출된, 이 스타일을 "리캐치답게" 만드는 핵심 특질입니다.

| # | 원칙 | 데이터 근거 | 적용 규칙 |
|---|---|---|---|
| 1 | **인사말 버리고 문제로 직진** | 인사말 도입 0건. 첫 문장이 곧 독자 고민 | 첫 줄에 반전·수치·고민·질문을 박는다 |
| 2 | **'우리'라고 말하는 동료 목소리** | "우리" 1,305회, 합니다체+해요체 혼용 | 가르치지 말고 옆에서 같이 고민하는 톤 |
| 3 | **독자를 전문가로 대우** | CRM·SEO·ICP·SDR·CTA·LTV 등 약어 그대로 사용 | 핵심 용어는 1회만 짧게 풀고 이후 그대로 |
| 4 | **주장 대신 증거** | 퍼센트 571회·배수 242회·금액 306회, 직접인용 약 1,400회 | 모든 주장에 숫자나 실명 인용을 붙인다 |
| 5 | **PAS 서사** (문제→심화→해법) | 하지만 424·특히 398·즉 286·예를 들어 243 | 고민 묘사 → 긴장 증폭 → 해법 착지 |
| 6 | **사고리더십 포지셔닝** | 본문 90%가 인사이트, 제품은 끝에 짧게 | 먼저 가르치고, 제품은 그 해법으로만 등장 |
| 7 | **강압 없는 초대형 CTA** | 클로징이 "~해보세요" 권유형 | "사세요"가 아니라 "한번 해보세요" |

## Part 2. 글 구조 템플릿 (Skeleton)

리캐치 글이 거의 예외 없이 따르는 7블록 흐름입니다.

```
[1] 후킹 도입부      → 인사말 X, 문제·장면·수치로 시작 (2~4단락, 중앙값 3단락)
[2] 💡 미리보기 박스  → 핵심 선요약 (선택, 핵심 글에 등장)
[3] "콘텐츠 순서" + 목차 → 항목이 본문 소제목과 1:1 연결 (필수)
[4] 본문 섹션 ×4~6   → 소제목 → 이미지 → 짧은 단락 (섹션당 이미지 1~2장)
[5] 종합/전환 섹션    → 사례·데이터·인용으로 가치 정리
[6] 📌 클로징 CTA     → 질문형 마무리 + 제품/콘텐츠 권유 1~2문장
[7] 마무리 이미지     → 인물 컷 또는 제품 화면
```

### 정량 규격 (중앙값 / 권장 범위)
| 요소 | 측정값 | 작성 기준 |
|---|---|---|
| 본문 분량 | 공백 제외 3,966자 | **2,800~5,300자**, 최소 2,800자 |
| 섹션(대주제) | 5개 | **4~6개** |
| 소제목(분기) | 평균 6개 | 섹션당 1~2개 |
| 이미지 | 7장 | **5~10장**, 섹션당 1~2장 |
| 단락 길이 | 159자 / 3.9문장 | 100~180자, 3~4문장 |
| 문장 길이 | 평균 30자대 | 짧게, 한 문장 한 메시지 |

### 도입부 후킹 4패턴 (실제 사용 빈도순)
1. **역설/반전형** — "CRM을 만드는 회사에도, CRM 입력을 대신하는 인력이 있었습니다."
2. **수치/긴박감형** — "세일즈 팔로업의 골든타임은 5분, 그런데…"
3. **공감/현장형** — "B2B에서 고객을 만나는 일은 생각보다 훨씬 구체적이고, 가끔은 꽤 고됩니다."
4. **질문/딜레마형** — "고객을 어디서 어떻게 만날 것인가."

> 도입부 **마지막 단락**에서 "이 글에서는 ~를 다룹니다" 식으로 **읽을 이유를 약속**한다.

## Part 3. 문체 가이드 (Voice & Tone)

### 종결어미
- **합니다체(격식)와 해요체(부드러움)를 섞어** 쓴다. 정보 전달 문장은 합니다체, 공감·권유 문장은 해요체.
- 단정·강조가 필요한 문장은 짧게 끊는다. (예: "이유는 단순했어요.")

### 어휘
- B2B 실무 용어·영문 약어를 그대로 쓴다. 단, **처음 등장 시 한 번만** 짧은 정의 박스로 풀어준다. (예: "할루시네이션이란? AI가 없는 내용을 사실처럼 만드는 현상")
- 'Re:' 워드플레이로 브랜드 자산을 콘텐츠명에 일관 삽입(Re:cast, Re:mail, Re:view 등). → **자사 버전으로 변형 가능** (브랜드 접두어 시스템화).

### 독자 관계 (engagement)
- "~하고 계신가요?", "~하지 않으신가요?" 같은 **질문으로 말 걸기** (본문 중 약 238회).
- "~해보세요" 권유 표현으로 행동 유도 (1,000회 이상).

### 증거·신뢰 장치
- **실명 인물 직접 인용("…")**을 적극 배치 — 영업담당자·고객·리더의 실제 발언.
- 숫자는 구체적으로: "효과가 좋다" ❌ → "고객사 60% 증가, 리드 3,000개" ⭕

### 시각·문장부호 장치
- 핵심구는 **굵게** (단락당 1~2개).
- 괄호 보조설명으로 친절히 풀기 (약 2,100회), 화살표(→)로 인과·결론 연결 (약 450회).
- 이모지 콜아웃을 의미별로 일관 사용: 💡 인사이트 / 📌 중요안내 / → 결론 / ✅ 체크 / 1️⃣2️⃣3️⃣ 단계.

## Part 4. 이미지 가이드

| 항목 | 기준 |
|---|---|
| 장수 | 5~10장 (중앙값 7) — 네이버 권장 "글 3줄+사진" 반복 패턴과 일치 |
| 밀도 | 섹션(대주제)당 1~2장 |
| 위치 | 새 섹션 시작 직후 대표컷 + 단락 사이 보조컷 + 맨 끝 마무리컷 |
| 형식 | 시연 화면 / 인물 / 도식 / 비교표. 캡션 73%, alt 74% 채움 |
| 캡션·alt | "무엇을 보여주는지" 설명형으로. **캡션에 핵심 키워드 1회 포함** (네이버 SEO) |

> **네이버 추가 팁:** 2026 알고리즘은 텍스트+이미지를 넘어 **동영상에 가산점**을 준다. 네이버 자체 영상이나 클립(숏폼, 30초~3분)을 1개 삽입하면 D.I.A+ 체류 점수에 유리.

## Part 5. 재·쓰·비(材·스타일·秘) 프롬프팅 프레임워크

리캐치가 공개한 **AI 콘텐츠 3요소**. writer 에이전트 프롬프트 설계에 그대로 매핑됩니다.

| 요소 | 의미 | 파이프라인 적용 |
|---|---|---|
| **재 (재료)** | 좋은 글감을 입력하라 | researcher 에이전트가 수집한 1차 자료·데이터·인용·사례를 충분히 투입. AI에게 빈손으로 "써줘" 금지 |
| **쓰 (스타일)** | 참고할 레퍼런스를 붙여 넣어라 | 이 가이드 + 우수 published 글 2~3편을 스타일 레퍼런스로 프롬프트에 첨부 |
| **비 (비밀)** | 나만의 기획안을 저장하라 | 글마다 사람이 만든 **기획안(관점·메시지·타깃·구조)**을 먼저 작성해 AI에 주입. 이것이 '좋은 글'의 핵심 |

> 리캐치의 결론: **도구(Claude)보다 중요한 건 '재료와 기획'이다.** AI는 잘 쓴 글을 만들지만, 좋은 글은 사람의 기획에서 나온다.

## Part 6. 네이버 최적화 — C-Rank & D.I.A+ 적용

리캐치 블로그(recatch.cc)는 구글 SEO 기반 워드프레스다. **네이버로 옮길 때**는 아래 두 알고리즘에 맞춰 변환한다.

### C-Rank (채널 신뢰도 = "이 블로그가 이 분야 전문인가")
- **한 주제(CS·CX·B2B)에 집중** 발행 → 전문성 누적. 잡블로그화 금지.
- 꾸준한 발행 주기 유지(리캐치는 격주 수요일 뉴스레터). 오래 운영된 채널일수록 유리.
- 사고리더십 콘텐츠(트렌드·방법론)가 C-Rank 전문성 신호로 직결.

### D.I.A+ (문서 품질 = "검색 의도 충족 + 독창적 경험 + 체류")
- **검색 의도 적중**: 제목·소제목을 실제 검색 질문형으로 (리캐치 소제목 19%가 질문형).
- **독창적 경험·데이터**: 자체 수치, 실명 사례, 1차 인용 → 복붙·일반론 문서와 차별화 (2026 알고리즘은 '진정성 있는 경험'에 트래픽 집중).
- **체류시간 방어**: 목차로 길 안내 + 짧은 단락 + 이미지/영상으로 스크롤 호흡 → "독자가 이 글 읽고 다른 검색 안 하게" 만든다.

### 2026 추가 변수 (GEO/AI 검색)
- 네이버가 하이퍼클로바X를 검색에 적용 → **AI가 먼저 해석하기 좋은 '정리된 정보 문서' 구조**가 유리.
- 리캐치 스타일의 목차·소제목·증거 기반 구조가 마침 여기에 부합한다.

### 네이버 변환 체크포인트 (recatch.cc → 네이버)
- 영문 약어는 첫 등장 시 **한글 병기** ("CRM(고객관계관리)") — 네이버 검색 키워드 매칭.
- `<nav>` HTML 목차 → 네이버 **자동목차/구분선+굵은 소제목**으로 구현.
- 제목·본문·이미지 캡션에 **타깃 키워드를 자연스럽게 분산** (과다 반복 금지).
- 네이버 클립/영상 1개 + (지역업종이면) 지도 블록 삽입.

## Part 7. 발행 전 체크리스트 (품질 검증 룰)

writer 산출물 통과 조건:

- [ ] **인사말로 시작하지 않는다** (문제·수치·질문 후킹) — *필수, 위반 시 reject*
- [ ] 목차 존재 + 본문 소제목과 1:1 연결 — *필수*
- [ ] 본문 ≥ 2,800자 (목표 3,500~5,000자)
- [ ] 대주제 4~6개, 그중 1개 이상 질문형 소제목
- [ ] 이미지 5~10장 + 섹션당 평균 1장 이상 + 캡션에 키워드
- [ ] 단락 평균 ≤ 180자, 단락당 굵게 1~2개
- [ ] 구체적 수치(%·배수·금액) 또는 실명 인용 1회 이상
- [ ] 도입부에 "이 글에서 다룰 내용" 약속 문장
- [ ] 클로징 = 질문형 마무리 + 강압 없는 권유형 CTA + 내부 링크
- [ ] 제품 언급은 본문 비중 ≤ 10%, 인사이트가 주(主)
- [ ] (네이버) 영문 약어 첫 등장 한글 병기, 영상/클립 1개, 키워드 자연 분산
- [ ] **'좋은 글' 게이트**: AI 초안에 사람이 만든 기획안(관점·메시지)이 반영됐는가?

## 요약 — 리캐치 스타일을 한 문장으로

**① 인사말 없는 문제 후킹 → ② 데이터·실명 인용의 증거 → ③ 사고리더십 인사이트 → ④ 강압 없는 제품 초대**,
이 4박자를 *전문가 동료의 목소리*와 *모바일 친화적 짧은 호흡*으로 풀어내되,
"AI가 쓴 잘난 글"이 아니라 **"사람이 기획한 좋은 글"**을 지향하는 방식.
