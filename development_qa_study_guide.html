1. AI 글자수 제한(Token Limit)과 우회 구조 설계

❓ 질문: "기존에 너가 문제라고 얘기한 게, 텍스트까지 이미지 메이킹 에이전트가 받아서 텍스트랑 이미지를 함께 출력해서 어셈블러 에이전트한테 넘기는 게 문제였다고 했는데 그게 왜 문제지? 어차피 최종 결과는 합쳐야 하잖아? 그리고 네이버 블로그 글이 왜 도입부만 출력되고 중간에 끊겼던 거야?"

모든 거대 언어 모델(Gemini, Claude, GPT 등)에는 "1회 답변당 출력할 수 있는 물리적인 최대 글자수 한도(Max Output Tokens)"가 존재합니다. 구글 Gemini API의 경우 이 한계치는 아무리 설정을 크게 늘려도 8,192 토큰(한글 기준 공백 포함 약 5,000자 내외)등입니다.

🚫 기존의 병목 구조 (글자수 초과로 인한 폭발)

작가 에이전트가 상세하게 쓴 6,600자 본문을 전달받아, 이미지 메이커 에이전트가 중간 마커 자리에 들어갈 대용량 HTML 카드 코드(CSS 포함) 7개를 생성하여 합쳐진 본문 전체(약 17,000자)를 AI에게 다시 출력하라고 요구했습니다. 이로 인해 API 최대 출력 한도(8,192 토큰)를 한참 초과하여 글이 잘리게 되었습니다.

⚙️ 개선된 스마트 우회 구조 (자바스크립트의 분할 조립)

[작가 AI] ➔ 본문 초안 작성 (6,600자) │ ▼ [이미지 메이커 AI] ➔ "본문은 적지 말고, 오직 이미지 카드 HTML 7개만 출력해 주세요." │ (출력량을 3,000자 수준으로 낮춰 잘림 방지) ▼ [일꾼 백엔드(JS)] ➔ AI가 준 카드를 메모리에 보관하고 마커 자리에 임시 번호표({{HTML_CARD_X}}) 배치 │ ▼ [조립가 AI(Assembler)] ➔ 번호표만 달린 가벼운 글을 받아 마크다운만 HTML로 변환 │ (가벼워서 절대 잘리지 않음) ▼ [최종 복원(JS)] ➔ 완성된 결과물의 번호표 자리에 보관했던 카드 HTML을 쏙쏙 끼워넣어 출력!

2. VERCEL 환경 변수(0과 1)와 로컬 파일 작동 원리

❓ 질문: "원래 VERCEL=1로 켜두었던 것은 실배포 환경을 똑같이 구현하기 위해서였잖아? 이걸 VERCEL=0으로 꺼두면 깃허브에 Merge 되었을 때 배포용 Render 서버가 로컬 파일을 읽으려고 해서 에러가 나지 않을까?"

전혀 걱정하실 필요 없습니다. .env 파일은 원래 깃허브에 올라가지 않기 때문입니다.

프로젝트 최상위의 .gitignore 규칙에 의해 render-worker/.env 파일은 개발자의 컴퓨터에만 존재하며, 깃허브 저장소에는 절대 반영되지 않습니다. 따라서 로컬에서 고친 VERCEL=0 설정은 깃허브 병합 시 배포 서버에 아무런 영향도 주지 않습니다.

💡 로컬 개발(VERCEL=0)과 실배포(VERCEL=1)의 완벽한 분리

로컬 개발 중에는 내 하드디스크의 가이드라인 지침서 파일들(image-maker-web.md 등)을 수정하면서 바로 테스트해야 하므로 VERCEL=0이 필요합니다. 이후 실제 Render 서비스 배포가 완료되면 Render 대시보드 환경변수 설정 창에 수동으로 VERCEL = 1을 직접 적어둡니다. 그러면 배포 서버는 정상적으로 깃허브에서 가이드라인을 다운로드하게 됩니다.

3. 배포와 Git Ignore, 그리고 데이터베이스 연동 보안

❓ 질문: "그러면 데이터베이스 주소(SUPABASE_URL)나 열쇠(API 키) 정보가 든 .env 파일도 깃허브에 안 올라갈 텐데, 실제 배포된 사이트에서는 어떻게 데이터베이스에 접속하고 통신할 수 있는 거지?"

보안 정보가 담긴 환경설정 파일을 소스코드 저장소(Git)에 올리는 것은 매우 위험합니다. 대신, 클라우드 호스팅 서비스들이 제공하는 "대시보드 환경변수 주입" 방식을 통해 해결합니다.

🔒 배포 서버 환경변수 수동 등록 공식:
  • Vercel (웹 게이트웨이): Vercel 설정 페이지 ➔ SUPABASE_SERVICE_ROLE_KEY, NEXT_PUBLIC_SUPABASE_URL 직접 등록
  • Render (백그라운드 일꾼): Render 설정 페이지 ➔ SUPABASE_URL, SUPABASE_SERVICE_ROLE_KEY, VERCEL=1 직접 등록

이렇게 수동으로 한 번만 꽂아두면, 깃허브에는 소스코드가 안전하게 공유되면서 각 배포 서버 컴퓨터들은 클라우드가 제공하는 비밀 키를 바탕으로 작동하게 됩니다.

4. dist 폴더의 정체와 안전한 삭제 이유

❓ 질문: "너 render-worker/dist 폴더 삭제해도 되겠어? 큰일 날 것 같은데. 이게 정확히 무슨 역할이길래 이렇게 과감하게 삭제한 거야?"

dist 폴더는 영구 보관용 소스 폴더가 아니라, 언제든 완전히 지워도 안전한 "임시 번역본 보관창고"입니다.

우리가 소스코드를 짤 때는 타입스크립트(.ts)를 쓰지만, 컴퓨터 엔진은 순수 자바스크립트(.js) 파일만 실행할 수 있습니다. 우리가 빌드 명령(npm run build)을 내릴 때 컴파일러가 원본 소스를 변환해서 새로 뱉어내 저장하는 장소가 바로 dist(Distribution) 폴더입니다.

원본 소스코드(index.ts와 src/ 폴더)가 그대로 살아있으므로 dist는 언제든 필요하면 0.5초 만에 처음부터 끝까지 완전 재창조가 가능합니다. 로컬 개발 중에는 실시간 실행 도구가 메모리에서 바로 번역해 돌려주기 때문에 이 폴더가 필요 없으며, 오히려 구버전 컴파일 캐시 파일이 고여있으면 버그를 유발하므로 자주 비우고 청소해 주는 것이 훨씬 이롭습니다.

5. 빌드 공정의 핵심: 압축 및 패키징 (Minification & Bundling) 상세

❓ 질문: "빌드 맨 마지막 단계에서 압축 및 패키징을 한다며? 이전 설명에는 왜 그 내용이 빠져있어? 구체적으로 어떻게 최적화되는지 알려줘."

컴파일러가 코드를 자바스크립트로 단순히 변환한 후, 실제 웹 사이트 배포에 적합하도록 코드를 극단적으로 다이어트시키는 작업이 바로 압축(Minification)과 패키징(Bundling)입니다.

① 패키징 (Bundling - 하나로 묶기)

개발 시 관리 효율성을 위해 쪼개둔 수십 개의 파일들(route.ts, agentRunner.ts 등)을 번들러가 관계 분석 후 단 하나의 통합 자바스크립트 파일(`main.js` 등)로 합쳐, 인터넷 다운로드 연결 횟수를 줄여 로딩 속도를 향상시킵니다.

② 압축 (Minification - 빈칸 및 여백 제거)

개발자가 가독성을 위해 남겨둔 줄바꿈(Enter), 여백(Space), 주석, 그리고 길게 작성한 변수명을 여백 제거 및 문자 압축 방식(예: 긴 변수명을 a로 치환)을 통해 극단적으로 경량화합니다. 이로 인해 소스코드 전체 크기가 70% 이상 가볍게 다이어트됩니다.

6. 더 근본적인 아키텍처 해결책: 데이터 구조의 원자화 (JSON 배열 설계)

❓ 질문: "이렇게 HTML 주석 기호를 활용하는 AI의 눈치(학습 과정)에 의존하는 방법 외에, 시스템적으로 에러를 0%로 만드는 근본적인 프로그래밍적 해결법은 없었어?"

가장 근본적이고 수학적으로 에러 확률이 0%인 설계 기법은 글을 단일 문자열로 취급하지 않고 "원자 단위의 JSON 부품 목록(Array) 데이터 구조"로 변환하여 처리하는 것입니다.

🛠️ 원자화 설계 동작 방식

작가 AI가 글을 쓸 때 하나의 텍스트 본문 대신, 각 요소가 구별된 JSON 리스트 데이터를 출력하도록 지시합니다. 그 뒤 자바스크립트 프로그램(일꾼)이 이 배열을 돌며 paragraph는 조립가 AI에게 단독 전달하고, image_request는 이미지 메이커 AI에게 던져 부품별 HTML 결과물을 따로 받아온 뒤, 최종 결합합니다.

⚠️ 그럼에도 이번 프로젝트에서 원자화 대신 주석 기호 우회를 선택한 이유

AI가 각 단락을 조각내어 개별적으로 전달받게 되면, 바로 위 문단에서 자신이 서술한 흐름을 읽지 못해 같은 단어를 무한 반복하거나 앞뒤 문맥이 꼬이는 글의 품질 하락 현상이 발생합니다. 또한 기존의 마크다운 기반 뼈대를 JSON 배열 구조로 다 뜯어고치려면 소모되는 개발 자원이 너무 큽니다. 따라서 단일 글 구조를 유지하되 AI가 무시하는 속성을 지닌 HTML 주석 기호(<!-- -->)를 안전장치로 배치하여 우회한 것이 최적의 실무 대안이 되었습니다.

7. 추론(Thinking) 토큰 병목 현상과 선택적 차단 해결법

❓ 질문: "생각을 안 하면 글의 퀄리티가 떨어지는 거 아니야? 그리고 API의 글자 토큰 한계 수치(maxOutputTokens)를 그냥 대폭 늘려버리면 간단히 해결될 일인데 왜 늘리지 않고 복잡한 다른 방식을 사용한 거지?"

구글 Gemini API 서버가 공식적으로 제공하는 1회 답변당 최대 출력 한계(maxOutputTokens)는 8,192 토큰이 물리적 상한선(Hard Limit)입니다. 이 상한선은 임의로 늘릴 수 없으며, 추론 생각 토큰(Thinking Tokens)도 이 한도 내에 차감됩니다. 입력 데이터가 무거워질수록 AI는 약 7,600자의 공간을 생각하는 데 소모해 버리며, 실제 답변을 적을 한도가 500자 미만으로 남아 글이 도중에 잘릴 수밖에 없었습니다.

⚙️ 선택적 생각 차단 (Selective Thinking Disable) 해결책

글의 퀄리티 저하를 막기 위해, 기획과 조립 각 단계의 특성을 고려하여 생각 기능을 필요한 단계에만 골라서 켜고 끄는(On/Off) 방식을 적용했습니다.

  • 1단계(리서치) 및 2단계(창작 작가) [Thinking On]: 아이디어를 짜내고 논리적인 글을 창작해야 하는 핵심 단계이므로 생각 기능을 유지하여 최고 수준의 글 퀄리티를 유지합니다.
  • 2.5단계(카드 메이커) 및 3단계(HTML 조립가) [Thinking Off]: 글을 HTML에 맞춰 감싸거나 디자인 템플릿에 맞추는 기계적 포맷팅 공정이므로 생각 예산을 차단(thinkingBudget: 0)하여 8,192자 공간 전체를 순수하게 HTML 본문 생성에만 투입하도록 하여 잘림 현상을 완벽히 차단했습니다.
