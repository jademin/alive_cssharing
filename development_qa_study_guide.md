## 1. AI 글자수 제한(Token Limit)과 우회 구조 설계

### ❓ 질문
> *"기존에 너가 문제라고 얘기한 게, 텍스트까지 이미지 메이킹 에이전트가 받아서 텍스트랑 이미지를 함께 출력해서 어셈블러 에이전트한테 넘기는 게 문제였다고 했는데 그게 왜 문제지? 어차피 최종 결과는 합쳐야 하잖아? 그리고 네이버 블로그 글이 왜 도입부만 출력되고 중간에 끊겼던 거야?"*

### 💡 핵심 답변 및 원리
모든 거대 언어 모델(Gemini, Claude, GPT 등)에는 **"1회 답변당 출력할 수 있는 물리적인 최대 글자수 한도(Max Output Tokens)"**가 존재합니다. 구글 Gemini API의 경우 이 한계치는 아무리 설정을 크게 늘려도 **8,192 토큰(한글 기준 공백 포함 약 5,000자 내외)**입니다.

#### 🚫 기존의 병목 구조 (글자수 초과로 인한 폭발)
1. **작가 에이전트:** 본문 글을 아주 상세하게 **6,600자** 작성했습니다.
2. **이미지 메이커 에이전트:** 본문 6,600자를 읽고 중간의 `[IMAGE]` 마커에 들어갈 HTML 카드 디자인 코드(카드 1개당 약 1,500자의 무거운 CSS 인라인 코드) 7개를 생성합니다. 이 카드 코드들과 원래 본문을 **합쳐서 전체 본문(약 17,000자)을 다시 화면에 출력**해야 했습니다.
3. **결과:** AI의 출력량이 **17,000자(약 12,000 토큰)**에 달했기 때문에, API 물리 한도(8,192)를 한참 초과하여 **도입부와 카드 한두 개만 그린 채 출력이 뚝 끊기는 참사(Truncation)**가 났던 것입니다.

#### ⚙️ 개선된 스마트 우회 구조 (자바스크립트의 분할 정복)
AI에게 무거운 본문을 그대로 베껴 쓰게 시키지 않고, 컴퓨터 연산(자바스크립트)이 결합 업무를 가로채도록 가볍게 공정을 개편했습니다.

* **[개선] 이미지 메이커 AI (Step 2.5):**
  * 더 이상 6,600자의 본문을 복사해서 출력하지 않습니다.
  * 오직 교체될 **HTML 이미지 카드 7개**의 코드만 독립적으로 출력합니다 (약 4,000자 분량).
  * 일꾼(Render) 프로그램이 이 카드 7개를 가로채어 메모리에 따로 보관합니다.
* **[개선] 일꾼의 번호표 치환 (JS Intervention):**
  * 일꾼 프로그램이 원본 초안(6,600자)의 `[IMAGE: ...]` 자리에 임시로 번호표(`<!-- HTML_CARD_X -->`)를 붙여서 가벼운 본문(6,600자)을 만듭니다.
  * 이 과정에서 `{{HTML_CARD_X}}`와 같은 중괄호 기호는 AI가 템플릿 변수로 착각해 발작을 일으키는 원인이 되므로, 표준 HTML 주석 형태인 `<!-- HTML_CARD_X -->`로 교체하여 AI가 해석하지 않고 복사만 하도록 안전하게 유도했습니다.
* **[개선] 조립가 에이전트 (Step 3):**
  * 어셈블러는 거대한 HTML 카드가 제거되고 번호표만 달린 6,600자짜리 가벼운 글을 받아 HTML로 변환합니다.
  * 무거운 HTML 카드 코드가 없기 때문에 어셈블러의 출력도 아주 가볍게 끝납니다 (글자 수 한도에 걸리지 않음).
* **[개선] 번호표 복원 (JS Restore):**
  * 어셈블러가 변환를 마치고 돌려준 최종 HTML에서, 일꾼 프로그램이 번호표(`<!-- HTML_CARD_X -->`)를 찾아 아까 따로 보관해 두었던 **진짜 HTML 이미지 카드 코드**로 쏙쏙 끼워 넣습니다.

## 2. VERCEL 환경 변수(0과 1)와 로컬 파일 작동 원리

### ❓ 질문
> *"원래 VERCEL=1로 켜두었던 것은 실배포 환경을 똑같이 구현하기 위해서였잖아? 이걸 VERCEL=0으로 꺼두면 깃허브에 Merge 되었을 때 배포용 Render 서버가 로컬 파일을 읽으려고 해서 에러가 나지 않을까?"*

### 💡 핵심 답변 및 원리
**전혀 걱정하실 필요 없습니다. `.env` 파일은 원래 깃허브에 올라가지 않기 때문입니다.**

1. **Git Ignore (깃허브 업로드 차단):**
   * 프로젝트 최상위의 `.gitignore` 규칙에 의해 `[render-worker/.env]` 파일은 개발자의 컴퓨터에만 존재하며, 깃허브 저장소에는 절대 반영되지 않습니다.
   * 따라서 내 컴퓨터에서 개발 편의를 위해 `VERCEL=0`으로 고친 설정은 깃허브 병합 시 배포 서버에 아무런 영향도 주지 않습니다.
2. **로컬 개발의 필요성 (VERCEL=0):**
   * `VERCEL=1` 상태에서는 내가 내 컴퓨터 하드디스크에서 가이드라인 지침서 파일(`image-maker-web.md` 등)을 아무리 수정해도, 일꾼 서버가 내 컴퓨터가 아닌 **깃허브 서버에 실시간 접속해서 옛날 파일**을 가져옵니다.
   * 로컬에서 마음껏 규칙서를 수정하고 바로바로 테스트하려면 일꾼이 내 하드디스크 파일을 읽어야 하므로 로컬에서는 `VERCEL=0` 상태여야 합니다.
3. **실배포 환경 (VERCEL=1):**
   * 나중에 실제 서비스용 웹 서버를 켤 때는 Render.com 홈페이지의 관리자 설정 페이지에서 직접 수동으로 **`VERCEL = 1`** 환경변수를 주입해 줍니다.
   * 그러면 배포 서버는 정상적으로 깃허브에서 최신 파일을 읽어가며 동작하게 됩니다.

## 3. 배포와 Git Ignore, 그리고 데이터베이스 연동 보안

### ❓ 질문
> *"그러면 데이터베이스 주소(SUPABASE_URL)나 열쇠(API 키) 정보가 든 .env 파일도 깃허브에 안 올라갈 텐데, 실제 배포된 사이트에서는 어떻게 데이터베이스에 접속하고 통신할 수 있는 거지?"*

### 💡 핵심 답변 및 원리
소스코드 파일 내부에 데이터베이스 주소나 비밀 열쇠를 적어서 업로드하는 것은 해킹의 위험이 있어 절대 금지되는 방식입니다. 대신 업계 표준으로 사용하는 **"클라우드 환경변수 수동 주입"** 방식으로 사용하는 가상 키들을 꽂아서 연동합니다.

```
[로컬 환경 작동]
내 컴퓨터의 파일(.env) ➔ Supabase 클라우드 데이터베이스 접속 (동작 완료)

[배포 환경 작동]
1. 코드는 .env 파일 없이 안전하게 깃허브를 통해 Vercel / Render로 배포됩니다.
2. Vercel 및 Render 홈페이지 화면의 [Environment Variables] 설정창에 접속합니다.
3. 우리가 발급받은 데이터베이스 주소와 마스터 키를 수동으로 입력하여 등록해 줍니다.
4. 배포된 서버 컴퓨터들은 파일 없이도 이 설정값을 읽어 Supabase 데이터베이스에 안전하게 접속합니다.
```

## 4. `dist` 폴더의 정체와 안전한 삭제 이유

### ❓ 질문
> *"너 render-worker /dist 폴더 삭제해도 되겠어? 큰일 날 것 같은데. 이게 정확히 무슨 역할이길래 이렇게 과감하게 삭제한 거야?"*

### 💡 핵심 답변 및 원리
**`dist` 폴더는 영구 보관용 소스 폴더가 아니라, 일종의 "컴파일 임시 번역본 보관창고"입니다. 언제든 삭제해도 100% 안전합니다.**

1. **배포용 자바스크립트 변환기:**
   * 우리가 짠 진짜 원본 코드는 타입스크립트(`.ts`) 형식입니다. 하지만 Node.js 컴퓨터 엔진은 순수 자바스크립트(`.js`) 파일만 읽을 수 있습니다.
   * 프로젝트 빌드 명령어(`npm run build` 혹은 `tsc`)를 입력하면, 번역기가 `index.ts` 등 진짜 소스코드를 전부 자바스크립트로 한 바퀴 번역해서 **`dist(Distribution)` 폴더에 새로 뱉어냅니다.**
2. **삭제해도 되는 이유:**
   * 진짜 소스코드(`index.ts`, `src/` 폴더)는 그대로 보존되어 있기 때문에, `dist` 폴더는 언제든 다시 빌드하면 0.5초 만에 완벽하게 새로 복원됩니다.
   * 로컬 개발 단계(`npm run dev`)에서는 `ts-node`라는 도구가 실시간으로 원본 소스를 머릿속에 기억하며 실행하므로 `dist` 폴더가 전혀 필요하지 않습니다.
   * 오히려 과거 빌드 시점에 생긴 낡은 번역본(`.js`)이 폴더 안에 고여 있으면, 컴퓨터가 최신 소스코드를 두고 낡은 컴파일본을 먼저 로딩하는 경로 해석 버그가 나므로 **싹 비우고 청소해 두는 것이 훨씬 유익**합니다.

## 5. 빌드 공정의 핵심: 압축 및 패키징 (Minification & Bundling) 상세

### ❓ 질문
> *"빌드 맨 마지막 단계에서 압축 및 패키징을 한다며? 이전 설명에는 왜 그 내용이 빠져있어? 구체적으로 어떻게 최적화되는지 알려줘."*

### 💡 핵심 답변 및 원리
컴파일러가 코드를 단순히 다른 언어로 바꾸는 작업을 한 후, 실제 웹 사이트 배포에 적합하도록 코드를 극단적으로 다이어트시키는 작업이 바로 **압축(Minification)**과 **패키징(Bundling)**입니다.

#### ① 패키징 (Bundling - 하나로 묶기)
* **왜 하나요?**
  * 개발을 할 때는 유지보수를 편하게 하기 위해 기능별로 수십, 수백 개의 파일(`route.ts`, `agentRunner.ts`, `aiConfig.ts` 등)로 쪼개서 작업합니다.
  * 하지만 사용자의 웹 브라우저가 사이트에 접속할 때 이 파일 수백 개를 제각각 호출해서 다운로드받으려면 로딩 속도가 끔찍하게 느려집니다.
* **어떻게 하나요?**
  * 번들러(Webpack, Vite 등)가 수많은 소스코드 파일들의 연결 고리를 분석하여 **단 하나의 파일(예: `main.js`)로 결합하고 용접**해 줍니다. 

#### ② 압축 (Minification / Uglification - 군더더기 없애기)
* **왜 하나요?**
  * 개발자가 읽기 편하게 작성해 둔 엔터(줄바꿈), 띄어쓰기(여백), 한글 주석, 긴 변수 이름 등은 컴퓨터에겐 쓸데없이 용량만 차지하는 데이터 쓰레기입니다.
  * 이때 변수의 활동 범위(Scope) 개념을 사용하여 서로 다른 함수 영역에 선언된 동일한 이름의 단축 변수(`const a`)들은 중복 충돌 없이 메모리에서 완벽히 독립적으로 처리되도록 안전하게 번역됩니다.
* **어떻게 하나요?**
  * 최적화 도구(SWC, Terser 등)가 실행 코드에서 **모든 주석, 빈 줄, 여백을 강제로 지워버리고**, 긴 변수나 함수 이름을 알파벳 한 글자로 단축해 버립니다.
  * *예시 (원본):*
    ```javascript
    // 활성화된 API 키를 덮어씌웁니다.
    const resolvedApiKeyOverride = task.api_key;
    ```
  * *예시 (압축 후):*
    ```javascript
    const a=t.api_key;
    ```
  * 이 공정을 거치면 전체 소스코드 파일의 크기가 **70% 이상 가볍게 다이어트**되어, 사용자의 스마트폰 브라우저에 눈 깜짝할 사이에 로드될 수 있게 만듭니다.

## 6. 더 근본적인 아키텍처 해결책: 데이터 구조의 원자화 (JSON 배열 설계)

### ❓ 질문
> *"이렇게 HTML 주석 기호를 활용하는 AI의 눈치(학습 과정)에 의존하는 방법 외에, 시스템적으로 에러를 0%로 만드는 근본적인 프로그래밍적 해결법은 없었어?"*

### 💡 핵심 답변 및 원리
가장 근본적이고 수학적으로 에러 확률이 0%인 설계 기법은 글을 단일 문자열로 취급하지 않고 **"원자 단위의 JSON 부품 목록(Array) 데이터 구조"**로 변환하여 처리하는 것입니다.

#### 🛠️ 원자화 설계 동작 방식
1. **JSON 구조화 출력:** 작가 AI가 글을 쓸 때 하나의 텍스트 본문 대신, 각 요소가 구별된 JSON 리스트 데이터를 출력하도록 지시합니다.
   ```json
   [
     { "type": "paragraph", "content": "점심시간 직전, 병원 로비는..." },
     { "type": "image_request", "description": "과부하된 데스크 스태프 일러스트" }
   ]
   ```
2. **독립 루프 연산:** 자바스크립트 프로그램(일꾼)이 이 배열을 돌며 `paragraph`는 조립가 AI에게 단독 전달하고, `image_request`는 이미지 메이커 AI에게 던져 부품별 HTML 결과물을 따로 받아온 뒤, 최종 결합합니다.
3. **완전 결합:** 돌아온 온전한 HTML 조각들을 자바스크립트가 문자열 병합 연산(`+`)을 통해 최종 결합합니다.

#### ⚠️ 그럼에도 이번 프로젝트에서 원자화 대신 주석 기호 우회를 선택한 이유
* **AI의 문맥 상실 (Context Loss):** AI가 각 단락을 조각내어 개별적으로 전달받게 되면, 바로 위 문단에서 자신이 서술한 흐름을 읽지 못해 **같은 단어를 무한 반복하거나 앞뒤 문맥이 꼬이는 글의 품질 하락 현상**이 발생합니다.
* **공수의 합리성:** 기존에 마크다운 파일 기반으로 가이드 및 프롬프트가 모두 결합되어 있는 상태에서 전체 뼈대를 JSON 배열 구조로 다 뜯어고치려면 소모되는 개발 자원이 너무 큽니다.
* 따라서 AI의 글 흐름을 온전히 묶어 다루기 위해 단일 글 구조를 유지하되, AI가 무시하는 속성을 지닌 **HTML 주석 기호(`<!-- -->`)를 안전장치로 배치하여 영리하게 우회한 것**이 최적의 실무 대안이 되었습니다.

## 7. 추론(Thinking) 토큰 병목 현상과 선택적 차단 해결법

### ❓ 질문
> *"생각을 안 하면 글의 퀄리티가 떨어지는 거 아니야? 그리고 API의 글자 토큰 한계 수치(maxOutputTokens)를 그냥 대폭 늘려버리면 간단히 해결될 일인데 왜 늘리지 않고 복잡한 다른 방식을 사용한 거지?"*

### 💡 핵심 답변 및 원리

#### ① 출력 토큰 한계를 더 늘릴 수 없는 이유 (API의 물리적 한계)
* 구글 Gemini API 서버가 공식적으로 제공하는 1회 답변당 **최대 출력 한계(maxOutputTokens)는 8,192 토큰이 우주의 상한선(Hard Limit)**입니다. 
* 이는 우리가 개발하는 프로그램 코드로 1만자, 2만자로 늘리고 싶다고 해서 늘릴 수 있는 성질의 것이 아닙니다. 구글 시스템 자체가 그 이상을 뱉어주지 않습니다.
* 설상가상으로 Gemini 2.5 Flash 모델이 답변 도중 수행하는 **"추론 생각 토큰(Thinking Tokens)"**마저도 이 8,192 한도 내에 합산되어 차감됩니다. 
* 입력 데이터가 무거워질수록 AI는 약 7,600자의 공간을 혼자 생각하는 데 소모해 버리며, 이로 인해 실제 답변을 적을 물리적 한도는 겨우 500자도 남지 않게 되어 글이 도중에 잘릴 수밖에 없었던 것입니다.

#### ② 선택적 생각 차단 (Selective Thinking Disable) 해결책의 작동 원리
글의 퀄리티 저하를 막기 위해, 기획과 조립 각 단계의 특성을 고려하여 **생각 기능을 필요한 단계에만 골라서 켜고 끄는(On/Off) 방식**을 적용했습니다.

* **1단계(리서치) 및 2단계(창작 작가):**
  * 머리를 굴려 아이디어를 짜내고 논리적인 글을 창작해야 하는 핵심 단계이므로 **생각 기능을 정상 가동(Thinking On)**하여 최고 수준의 글 퀄리티를 유지합니다.
* **2.5단계(카드 메이커) 및 3단계(HTML 조립가):**
  * 이 단계는 새로운 창작을 하는 것이 아니라, 앞서 만들어진 글을 읽고 HTML 규칙에 맞춰 감싸거나 디자인 템플릿에 맞추는 **"기계적 포맷팅 공정"**입니다.
  * 여기서는 깊은 추론이 개입해 봤자 퀄리티 향상에 아무런 도움이 되지 않으므로, 호출 매개변수에 `disableThinking: true`를 넘겨 **생각 기능을 완전히 차단(Thinking Off)**합니다.
  * 그 결과, AI는 단 1토큰의 생각 낭비도 없이 **8,192자 전체 공간을 100% 온전히 HTML 카드 생성과 본문 렌더링에만 투입**할 수 있게 되어 잘림 현상이 완벽하게 영구 차단됩니다.
